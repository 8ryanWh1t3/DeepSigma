The Gap

Most organizations can identify risk. Few can trace a finding through review, response, and verification.

📋
Findings Without Provenance
Audit results, scan outputs, and assessment findings land in spreadsheets with no chain of evidence back to the original observation.
🔄
Recurring Problems
The same issues surface year after year because there is no durable record of what was fixed, who approved it, and whether the fix held.
🤝
Handoff Friction
Assessors hand findings to reviewers who hand decisions to implementers. Context is lost at every boundary.
⏳
Stale Risk Decisions
Risk acceptances persist long after the conditions that justified them have changed. No mechanism flags contradictions.
🔍
No Verification Loop
Patches are assumed effective. Corrections are marked complete without retest. Nobody confirms the fix survived real conditions.
📉
Lagging Visibility
Leadership sees last quarter's report, not the current posture. Decisions are made on stale data.

The CERPA Risk Lifecycle

A common operational language for risk management

CERPA provides a common operational language for risk management. Each phase moves risk from identification through evidence, review, corrective action, and authorized execution. Every decision carries its evidence chain.

CLAIM
Identifies the risk
EVENT
Captures the evidence
REVIEW
Determines significance
PATCH
Defines the response
APPLY
Authorizes execution
Standardizes how risk is identified, reviewed, and acted on. Accelerates movement from finding to decision to action. Creates traceability across assessment, response, and execution.

Data Model

Every CERPA record carries structured fields across all five phases

A CERPA record is a single unit of governance memory. It tracks one finding from initial assertion through verified correction. Fields are grouped by phase, and every record carries a confidence score reflecting the completeness of its evidence chain.

// A single CERPA record
{
  "id": string, // unique record identifier
  "source": string, // originating system, report, or assessment
  "created": ISO 8601,
  "confidence": 0.00 – 1.00, // evidence completeness score

  // ─── CLAIM ───
  "claim_id": string,
  "claim_text": string, // the asserted condition or requirement
  "claim_source": string, // framework, standard, or policy reference
  "claim_class": string, // finding classification (severity/type)
  "claim_refs": string[], // control, clause, or requirement references

  // ─── EVENT ───
  "event_id": string,
  "event_type": string, // observation method (interview, document, test, scan)
  "event_evidence": string, // what was actually observed
  "event_timestamp": ISO 8601,
  "event_observer": string, // who or what captured the evidence

  // ─── REVIEW (Risk Assessment) ───
  "review_id": string,
  "review_decision": string, // accept, mitigate, transfer, defer
  "review_rationale": string, // why this decision was made
  "review_authority": string, // who authorized the decision
  "review_severity": string, // critical, high, moderate, low
  "review_likelihood": string, // almost certain, likely, possible, unlikely, rare
  "review_impact": string, // assessed impact level
  "review_residual_risk": string, // expected risk level after treatment

  // ─── PATCH (Risk Treatment) ───
  "patch_id": string,
  "patch_action": string, // mitigate, control, transfer, accept, escalate
  "patch_description": string, // the corrective action taken
  "patch_owner": string, // responsible party for implementation
  "patch_deadline": ISO 8601, // due date
  "patch_status": string, // planned, in_progress, complete

  // ─── APPLY ───
  "apply_id": string,
  "apply_verified": boolean, // was the correction independently verified?
  "apply_method": string, // how verification was performed
  "apply_result": string, // pass, fail, partial
  "apply_durable": boolean // did the fix survive retest?
}

Confidence Scoring

Every CERPA record carries a confidence score (0.00 – 1.00) that reflects evidence completeness. The score is computed from weighted components — not a subjective judgment, but a mechanical measure of how much of the lifecycle has been filled in.

+0.20
Claim reference present
+0.20
Event evidence captured
+0.20
Review decision recorded
+0.20
Patch action documented
+0.20
Apply verification done
A score of 1.00 means every phase has evidence. A score of 0.40 means the record has a claim and an event but no review, no patch, and no verification — an open governance gap.

Risk Panel

Every CERPA record carries risk attributes that are populated during Review and refined through Patch and Apply. These fields make CERPA legible as an operational risk register.

Severity
Critical, High, Moderate, Low. Set during Review based on the potential consequence if the risk is realized.
Likelihood
Almost Certain, Likely, Possible, Unlikely, Rare. Derived from Event evidence and historical frequency.
Impact
Assessed during Review. Encompasses mission, financial, reputational, legal, and operational dimensions.
Confidence
0.00 – 1.00. Mechanical score reflecting evidence completeness across all five CERPA phases.
Owner & Due Date
Set during Patch. Names the accountable party and the deadline for treatment completion.
Residual Risk
Expected risk level after treatment is applied and verified. Only confirmed durable in the Apply phase.

Lifecycle Walkthrough

A single finding traced through all five phases

This walkthrough follows one governance record from initial observation to verified correction. The domain is deliberately generic — CERPA applies the same way whether the finding comes from an audit, a security assessment, a compliance review, or an operational inspection.

Phase 1 — CLAIM
An assessor identifies that access control procedures do not meet the requirements specified in the governing standard. The claim references a specific control (e.g., clause 9.2), classifies the finding as a nonconformity, and establishes the gap between the stated requirement and the observed state.
Artifacts: finding ID, control reference, classification, requirement text, assessor identity
Phase 2 — EVENT
The evidence that substantiates the claim is captured. Interviews with personnel confirm that access reviews are not performed on the required schedule. Document review reveals the access control policy was last updated 18 months ago. System logs show no evidence of periodic access recertification.
Artifacts: interview notes, document timestamps, system log excerpts, observation type, evidence quality assessment
Phase 3 — REVIEW
The finding is adjudicated by authorized decision-makers. They confirm the nonconformity, assess the impact as moderate, and determine that the corrective action must address both the policy gap and the missing technical controls. A remediation deadline is established and the decision is recorded with the identity of the authorizing official.
Artifacts: decision record, impact assessment, authorizing official, remediation deadline, rationale
Phase 4 — PATCH
The corrective action is implemented. The access control policy is updated to reflect current requirements. An automated access recertification process is deployed. Personnel are trained on the new procedure. Change control records capture who approved the changes, when they were implemented, and what evidence was used to justify the approach.
Artifacts: updated policy document, technical change records, training completion, change approval, implementation owner
Phase 5 — APPLY
The correction is independently verified. An assessor confirms that access reviews now occur on schedule, the updated policy is in force, and the automated recertification process has completed at least one cycle successfully. The finding is marked as resolved with durable evidence. The record enters institutional memory as a baseline for future assessments.
Artifacts: verification results, retest evidence, durability confirmation, memory graph update, confidence: 1.00
From finding → report → forget  to  claim → evidence → decision → correction → verification

What Coherence Measures

Completeness — what percentage of CERPA records have evidence across all five phases? Incomplete records represent governance gaps: findings without decisions, decisions without corrections, corrections without verification.
Consistency — do review decisions align with evidence? Do patch actions address the actual findings? Coherence detects contradictions: a risk accepted in one record that is being actively mitigated in another.
Durability — how many corrections survived independent verification? A patch that was never retested contributes less to coherence than one that was verified and confirmed durable.
Regression — are previously resolved findings reappearing? Regression alerts fire when a new claim matches a closed record, indicating the original correction did not hold. This is a direct measure of institutional memory quality.

Traditional vs. Governed

Traditional risk management records risk. CERPA governs risk through action and memory.

Design Principles

🔗
Evidence Chaining
Every phase links to the previous. No decision exists without the evidence that produced it.
🧠
Institutional Memory
Records persist across assessment cycles. Future assessments build on past findings instead of starting from scratch.
🔍
Contradiction Detection
Stale risk acceptances and conflicting decisions are surfaced automatically as the evidence base grows.
🔄
Regression Awareness
When a previously resolved finding reappears, the system flags it immediately. Patches that do not hold are identified.
📊
Real-Time Posture
Executive visibility comes from live data, not quarterly reports. Coherence scores reflect current state.
🌍
Domain Agnostic
The five phases are universal. Security, compliance, procurement, operations — any process that produces findings maps onto CERPA.
